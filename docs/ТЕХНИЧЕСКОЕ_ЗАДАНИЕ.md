## 1. Термины и сокращения

| Термин | Определение |
|--------|-------------|
| **SaaS** | ПО, предоставляемое как сервис по подписке |
| **Тенант (tenant)** | Изолированное рабочее пространство пользователя (компании) |
| **Соц-аккаунт (social account)** | Подключённый аккаунт/страница/канал в соцсети |
| **Драйвер (driver/adapter)** | Модуль интеграции с конкретной соцсетью |
| **Публикация (publication)** | Единичный акт размещения контента в соцсети |
| **Пост** | Контент (текст + медиа), подготовленный к публикации |
| **Воркер** | Фоновый процесс, выполняющий запланированные публикации |
| **JWT** | JSON Web Token |
| **RBAC** | Role-Based Access Control |
| **CDN** | Content Delivery Network |

## 2. Назначение и цели

### 2.1. Назначение
Веб-приложение для централизованного планирования, подготовки и автоматической публикации контента в несколько социальных сетей с разных аккаунтов.

### 2.2. Цели
1. Снизить трудозатраты на ведение нескольких соцсетей (один календарь на все каналы).
2. Обеспечить надёжность доставки публикаций (≥ 98%).
3. Дать масштабируемую SaaS-модель с биллингом и тарифами.
4. Заложить расширяемую архитектуру для добавления новых соцсетей без переработки ядра.

## 5. Требования к интеграциям (VK, Telegram)

### 5.1. ВКонтакте

**Авторизация:** VK OAuth 2.0 (Implicit/Code flow) с получением `access_token` и `refresh_token`.

**Области прав (scope):**
- `wall` — публикация на стене
- `photos` — загрузка фото
- `video` — загрузка видео
- `groups` — список управляемых сообществ
- `offline` (с учётом актуальной политики VK)

**Используемые методы API:**
| Операция | Метод |
|----------|-------|
| Список групп пользователя | `groups.get` (filter=admin) |
| Загрузка фото на стену | `photos.getWallUploadServer` → `photos.saveWallPhoto` |
| Загрузка видео | `video.save` |
| Публикация поста | `wall.post` (с `publish_date` для отложенной, без — для немедленной) |
| Удаление поста | `wall.delete` |
| Метрики поста | `wall.getById`, `stats` (для сообществ) |

**Требования:**
- Соблюдение лимитов API (429/ограничения по методам).
- Обработка `captcha` и `validation` (SMS) — для не-standalone-flow использовать серверный токен приложения.
- Хранение токенов в зашифрованном виде (см. раздел 10).
- Обновление `access_token` через `refresh_token` заранее (за 24ч до истечения).

### 5.2. Telegram

**Способ:** официальный Bot API. Бот создаётся через `@BotFather`, пользователь добавляет бота **администратором** в целевой канал с правом `post_messages`.

**Подключение канала (флоу):**
1. Пользователь в UI инициирует подключение → сервис генерирует уникальный одноразовый код.
2. Пользователь пишет боту команду `/start <code>`.
3. Бот связывает Telegram-пользователя с тенантом.
4. Пользователь добавляет бота админом в канал.
5. Сервис проверяет наличие прав бота в канале через `getChat`.

**Используемые методы:**
| Операция | Метод |
|----------|-------|
| Привязка чата | `getChat`, `getChatMember` |
| Отправка текста | `sendMessage` |
| Отправка фото | `sendPhoto` (multipart) |
| Отправка видео | `sendVideo` |
| Медиагруппа (альбом) | `sendMediaGroup` |
| Удаление/редактирование | `deleteMessage`, `editMessageText` |

**Ограничения:**
- Лимит файла Bot API: 50 МБ. Для больших — указать требование MTProto или загрузку по URL.
- Частота отправки: не более 30 сообщений/сек глобально, вводить собственные лимиты на пользователя.

### 5.3. Архитектура интеграций (паттерн Driver/Adapter)

```
┌──────────────────────────────────────────────┐
│           Publication Service (ядро)         │
│   формирует унифицированную задачу поста     │
└───────────────┬──────────────────────────────┘
                │  унифицированный интерфейс
   ┌────────────┼────────────┐
   ▼            ▼            ▼
┌──────┐   ┌────────┐   ┌──────────┐
│ VK   │   │Telegram│   │ Future:  │
│Driver│   │ Driver │   │ Meta, YT │
└──────┘   └────────┘   └──────────┘
```

**Интерфейс драйвера (TypeScript):**
```ts
interface SocialDriver {
  platform: Platform;
  authorize(tenantId, authCode): Promise<Account>;
  refreshToken(account): Promise<TokenSet>;
  validate(account): Promise<boolean>;          // жив ли доступ
  publish(account, post: UnifiedPost): Promise<PublishedResult>;
  delete(account, externalId): Promise<void>;
  fetchMetrics(account, externalId): Promise<Metrics>;
}
```

`UnifiedPost` — нормализованная модель контента (текст, медиа, ссылки); драйвер адаптирует её под специфику соцсети.

### 6.3. Поток публикации (sequence)

```
Пользователь → API: POST /posts {content, accounts[], scheduledAt}
   API:
     1) Валидация, проверка лимитов тарифа
     2) Сохранение Post + PostPublication (по одной на аккаунт) со статусом "queued"
     3) Для каждого PostPublication: job в очередь с delay = scheduledAt - now
     4) Ответ 201 с идентификаторами

Worker (по срабатыванию job):
     1) Достаёт PostPublication, ставит "in_progress"
     2) Получает свежий токен аккаунта (обновляет при необходимости)
     3) Через драйвер публикует контент
     4) При успехе → "published" + сохраняет externalId/URL
     5) При ошибке → ретрай (до 3, backoff), затем "failed" + уведомление
```

## 7. Модель данных

### 7.1. Основные сущности (ER-обзор)

```
Tenant 1──* User

Tenant 1──* Post 1──* PostPublication
```

### 7.2. SQL-схема (ключевые таблицы)

```sql
-- Пользователи
CREATE TABLE users (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         CITEXT UNIQUE NOT NULL,
  password_hash TEXT,
  name          TEXT,
  timezone      TEXT NOT NULL DEFAULT 'Europe/Moscow',
  locale        TEXT NOT NULL DEFAULT 'ru',
  is_email_verified BOOLEAN DEFAULT FALSE,
  totp_secret   TEXT,
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now()
);

-- Тенант (компания/рабочее пространство)
CREATE TABLE tenants (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  plan        TEXT NOT NULL DEFAULT 'free',  -- free, pro, business
  trial_ends_at TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- Связь пользователь-тенант + роль
CREATE TABLE tenant_members (
  tenant_id UUID REFERENCES tenants(id) ON DELETE CASCADE,
  user_id   UUID REFERENCES users(id) ON DELETE CASCADE,
  role      TEXT NOT NULL DEFAULT 'member', -- owner, member
  PRIMARY KEY (tenant_id, user_id)
);

-- Подключённые соц-аккаунты
CREATE TABLE social_accounts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID REFERENCES tenants(id) ON DELETE CASCADE,
  platform      TEXT NOT NULL,                 -- vk, telegram
  external_id   TEXT NOT NULL,                 -- id группы/канала
  display_name  TEXT,
  access_token_enc  BYTEA NOT NULL,            -- зашифровано
  refresh_token_enc BYTEA,
  token_expires_at TIMESTAMPTZ,
  metadata      JSONB,                         -- avatar, type, и т.п.
  status        TEXT NOT NULL DEFAULT 'active',-- active, expired, revoked
  created_at    TIMESTAMPTZ DEFAULT now(),
  UNIQUE (platform, external_id, tenant_id)
);

-- Пост (контент)
CREATE TABLE posts (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID REFERENCES tenants(id) ON DELETE CASCADE,
  author_id   UUID REFERENCES users(id),
  title       TEXT,
  body        TEXT NOT NULL,
  status      TEXT NOT NULL DEFAULT 'draft',   -- draft, scheduled, partial, done
  scheduled_at TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

-- Медиа поста
CREATE TABLE media_assets (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id   UUID REFERENCES posts(id) ON DELETE CASCADE,
  kind      TEXT NOT NULL,            -- image, video
  storage_key TEXT NOT NULL,          -- ключ в S3
  mime_type TEXT,
  size_bytes BIGINT,
  ord       INT DEFAULT 0
);

-- Планируемая публикация (пост × аккаунт)
CREATE TABLE post_publications (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id       UUID REFERENCES posts(id) ON DELETE CASCADE,
  social_account_id UUID REFERENCES social_accounts(id) ON DELETE CASCADE,
  status        TEXT NOT NULL DEFAULT 'queued', -- queued,in_progress,published,failed,cancelled
  scheduled_at  TIMESTAMPTZ NOT NULL,
  published_at  TIMESTAMPTZ,
  external_id   TEXT,                 -- id поста в соцсети
  external_url  TEXT,
  error_code    TEXT,
  error_message TEXT,
  attempts      INT DEFAULT 0,
  metrics       JSONB,                -- {views, likes, reposts, comments}
  UNIQUE (post_id, social_account_id)
);

-- Подписка/биллинг
CREATE TABLE subscriptions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID REFERENCES tenants(id) ON DELETE CASCADE,
  plan        TEXT NOT NULL,
  status      TEXT NOT NULL,          -- active, canceled, past_due
  current_period_end TIMESTAMPTZ,
  provider    TEXT,                   -- yookassa, stripe
  provider_customer_id TEXT,
  provider_subscription_id TEXT
);

CREATE TABLE payments (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  subscription_id UUID REFERENCES subscriptions(id),
  amount      NUMERIC(12,2) NOT NULL,
  currency    TEXT NOT NULL,
  status      TEXT NOT NULL,
  paid_at     TIMESTAMPTZ,
  provider_payment_id TEXT
);

-- Журнал уведомлений
CREATE TABLE notification_logs (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID REFERENCES users(id),
  channel    TEXT NOT NULL,           -- email, in_app
  type       TEXT NOT NULL,
  payload    JSONB,
  sent_at    TIMESTAMPTZ DEFAULT now(),
  status     TEXT
);

CREATE INDEX idx_pp_status_scheduled ON post_publications(status, scheduled_at);
CREATE INDEX idx_posts_tenant_scheduled ON posts(tenant_id, scheduled_at);
CREATE INDEX idx_sa_tenant ON social_accounts(tenant_id, platform);
```

### 7.3. Индексы и производительность
- Вышеуказанные индексы покрывают самые частые запросы (очередь, лента тенанта).
- Партиционирование `notification_logs` и `post_publications` по месяцам — при росте.

## 9. Нефункциональные требования

### 9.1. Производительность
- Отклик API p95 ≤ 300 мс на 95% запросов.
- Загрузка страницы (LCP) ≤ 2.5 с при 3G+.
- Воркер обработки очереди берёт задачу ≤ 5 сек после наступления scheduledAt.
- Целевая нагрузка на старте: 100 RPS API, 10 публикаций/сек.

### 9.2. Масштабируемость
- Горизонтальное масштабирование stateless-сервисов (API, воркеры).
- Read-реплики PostgreSQL при росте.
- BullMQ-очередь поддерживает шардинг и конкурентное исполнение.
- S3 — без ограничений по объёму.

### 9.3. Надёжность и доступность
- Целевой uptime: 99.9%.
- Бэкапы БД: ежедневно + PITR, хранение 30 дней.
- Идемпотентность публикаций: дедупликация по `(post_id, social_account_id)`.
- Graceful shutdown воркеров: завершение текущей задачи, перебалансировка очереди.

### 9.4. Совместимость
- Браузеры: последние 2 версии Chrome/Firefox/Safari/Edge.
- Адаптив: min 360 px ширина.

### 9.5. Локализация
- Интерфейс: русский (старт), английский (структура готова, i18n с первого дня).

## 10. Безопасность и соответствие

### 10.1. Аутентификация и авторизация
- Пароли — bcrypt (cost ≥ 12) или argon2id.
- JWT access (15 мин) + refresh (30 дней, httpOnly cookie, ротация).
- RBAC на уровне тенанта.

### 10.2. Хранение секретов
- Токены соцсетей — шифрование в покое (AES-256-GCM), ключ в vault (HashiCorp Vault / KMS).
- Секреты приложения — переменные окружения / vault, не в коде и не в БД.

### 10.3. Защита данных
- TLS 1.2+ на всех соединениях.
- Валидация входных данных (class-validator / zod), защита от XSS, SQL-инъекций (параметризованные запросы Prisma).
- CORS allow-list, CSP-заголовки.
- Защита от брутфорса: rate-limit на `/auth/*`, аккаунт-локап после 5 неудач.

### 10.4. Приватность и соответствие
- Сбор минимально необходимых данных.
- Пользовательское соглашение и политика конфиденциальности до начала использования.
- Хранение данных пользователей в выбранной юрисдикции (РФ/ЕС — по решению).
- Удаление аккаунта → каскадное удаление данных + отзыв токенов соцсетей в течение 30 дней.
- Соответствие 152-ФЗ (если РФ) / GDPR (если ЕС): согласие на обработку, право на удаление, экспорт данных.

### 10.5. Аудит
- Логирование критичных действий: подключение/отключение аккаунта, публикация, смена тарифа, вход.
- Хранение audit-логов ≥ 12 мес.

### 10.6. Безопасность медиа
- Антивирусная проверка загруженных файлов (ClamAV).
- Ограничение типов (mime whitelist) и размеров.
- Приватные URL с подписью для доступа к медиа.

## 12. Инфраструктура и развертывание

### 12.1. Окружения
| Окружение | Назначение | Данные |
|-----------|------------|--------|
| local | Разработка | Docker Compose, тестовые БД |
| dev | Общая для команды | изолированные тестовые данные |
| stage | Приёмка, тесты интеграций с реальным API (sandbox) | копия схемы prod |
| prod | Боевой | реальные данные |

### 12.2. CI/CD
- На каждый PR: линтер (ESLint), форматирование (Prettier), type-check, unit-тесты, build.
- На merge в `main`: автодеплой в dev; ручной апрув → stage → prod.
- Миграции БД (Prisma Migrate) — отдельным шагом с possible rollback.

### 12.3. Развёртывание
- Контейнеры Docker, оркестрация Kubernetes (prod) / Compose (dev).
- Ingress (Nginx) + TLS (Let's Encrypt / сертификаты).
- HPA для API и воркеров по CPU/RPS.
- secrets через Kubernetes Secrets / External Secrets + Vault.

### 12.4. Мониторинг и оповещения
- Метрики: Prometheus → Grafana (RPS, латентности, ошибки, длина очереди, % успешных публикаций).
- Логи: структурированный JSON → Loki/ELK.
- Трейсинг: OpenTelemetry → Jaeger (опционально).
- Ошибки фронт/бэк: Sentry.
- Алерты: падение % успешных публикаций < 95%, рост очереди, рост 5xx.

## 13. Тестирование

### 13.1. Уровни
| Уровень | Инструмент | Покрытие |
|---------|------------|----------|
| Unit | Jest / Vitest | Сервисы, утилиты ≥ 80% |
| Integration | Jest + Testcontainers | Репозитории, очереди, драйверы с мок-API |
| E2E | Playwright | Ключевые сценарии UI + API |
| Нагрузочное | k6 | Профиль 100 RPS, всплески 500 RPS |
| Безопасное | OWASP ZAP, Snyk (dep scan) | Перед релизом |

### 13.2. Ключевые E2E-сценарии (definition of test)
1. Регистрация → подключение VK → создание поста → немедленная публикация → статус «published».
2. Подключение Telegram-канала через бота → успешная валидация.
3. Планирование поста в 2 соцсети → публикация по расписанию → метрики собраны.
4. Имитация ошибки API соцсети → ретрай → финальный статус «failed» + уведомление.
5. Достижение лимита тарифа → блокировка создания поста с понятной ошибкой.

### 13.3. Тестовые данные и песочницы
- VK: standalone-приложение в режиме «тестирование».
- Telegram: отдельный тестовый бот + приватный канал.
